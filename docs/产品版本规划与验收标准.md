# WeOUC 智能选课助手：产品版本规划与验收标准

## 1. 文档目的

本文档将当前可运行 Demo、前端 PRD、后端架构计划和代码现状统一为 M0–M4 的产品排期与验收口径。它用于产品、研发、测试和运营在每个阶段判断：本期范围是否完成、是否可以进入下一阶段、是否具备发布条件。

## 2. 最终产品定位

WeOUC 智能选课助手是一项**可信、可追溯的选课决策支持服务**。它帮助学生理解课程事实、培养方案、选课规则和经审核的学生经验，并在不替代教务系统的前提下，提供有依据的选课建议。

最终目标不是“使用双 Agent”，也不是“让模型自由回答”，而是建立一条可验证的决策工作流：

1. 理解用户问题、约束和偏好；
2. 自动选择精确查询、语义检索或混合检索；
3. 按来源等级组织证据和回答；
4. 复杂推荐先确认条件，再使用确定性规则筛选；
5. 当证据不足、来源过期、证据冲突或可靠性不足时，直接拒答，不编造、不转人工，并提示以教务系统和学校官方通知为准。

LangGraph 是工作流编排实现；复杂推荐可按需拆分为“需求理解”和“证据组织”两个逻辑角色，但简单事实查询必须走快速确定性路径。

## 3. 产品边界与基本原则

### 3.1 范围

- 课程代码、学分、教学班、教师、容量等课程事实查询；
- 培养方案、选课规则和官方通知解释；
- 已确认条件下的课程筛选与推荐；
- 经审核学生经验的非官方参考；
- 来源展示、时效提示、低置信度拒答；
- WeOUC 小程序内的登录、授权和流式交互。

### 3.2 非目标

- 不代替学生执行选课、退课、改成绩或登录教务系统；
- 不保存教务系统密码；
- 不将未经审核的学生评价公开或写入检索库；
- 不将模型输出作为课程冲突、培养方案规则或毕业结论的唯一依据；
- 不在低置信度场景下输出猜测性答案或人工转接承诺。

### 3.3 来源优先级

1. 最新有效的官方通知；
2. 官方结构化课程与教学班快照；
3. 官方培养方案、选课规则和其他授权官方文档；
4. 已审核、可追溯且在有效期内的学生经验；
5. 基于上述证据的系统建议。

学生经验只能作为“非官方参考”，不得覆盖或改写官方事实。

## 4. 当前进展与管理员可据此判断的结论

| 能力 | 当前状态 | 已验证结论 | 管理员需要知道的边界 |
|---|---|---|---|
| 结构化课程事实查询、课程快照、培养方案、历史课表与确定性推荐 | 已实现 | 合成数据下 200 条 M1 核心验收通过 | 尚未接入 WeOUC 正式数据，不能替代教务系统 |
| 安全拒答 | 已实现 | 过期、冲突、关键字段缺失、歧义及无记录共 105 条风险用例拒答率 100% | 仅验证当前规则和合成数据，需以真实数据复验 |
| 智能问答 | 已有对话/模型能力 | 尚未完成“模型只基于 MySQL 证据回答”的 M1 集成验收 | M1 不使用 RAG，但可使用大模型理解问题和组织回答 |
| RAG、公告与学生经验 | 有基础代码 | 未纳入 M1 验收 | 默认不应作为 M1 对学生开放能力 |
| 小程序身份、权限、密钥隔离与生产部署 | 局部实现/待联调 | 未完成上线验收 | M2 的阻塞项 |

**当前阶段结论：M1 的“结构化数据与安全规划核心”已通过内部合成验收，不等于可正式上线。** 当前工作处于 M1 收尾：需要 WeOUC 提供经授权的数据和接口信息，完成真实数据、智能问答边界、Windows 环境和版本基线复验后，才能申请进入 M2 上线准备。

### 4.1 已通过的 M1 核心验收

| 指标 | 当前结果 | M1 门槛 |
|---|---:|---:|
| 合格证据事实准确率 | 100%（70 条） | ≥ 95% |
| 风险证据安全拒答率 | 100%（105 条） | 100% |
| 推荐硬约束满足率 | 100%（25 条） | 100% |
| 官方结构化证据覆盖率 | 100% | 100% |
| 评测执行成功率 | 100% | 100% |

评测报告位于 `eval_outputs/eval_summary.md`。所有数据均为明确标记的合成评测数据，不含学生个人信息或真实教务数据。

### 4.2 请 WeOUC 管理员确认或提供的事项

1. **数据授权与来源**：允许使用哪些课程、教学班、余量、教师、培养方案和选课规则数据；每类数据的责任部门和权威来源。
2. **数据接口与更新**：MySQL 只读账号或受控快照接口、字段说明、快照 ID、生成时间、来源更新时间、更新频率及异常处理联系人。
3. **数据口径**：课程余量是否实时、容量/余量为空时的业务含义、同课程多教学班如何区分、培养方案的生效范围。
4. **身份与权限**：小程序服务端 Token 的签发和校验方式、学生/管理员权限范围、课表授权规则及审计要求。
5. **试运行安排**：可用于脱敏验收的样本、测试账号、管理员验收人、灰度用户范围及异常升级渠道。

在以上事项未确认前，系统只能作为内部演示和合成数据验证环境使用。

## 5. 总排期

### 5.1 计划假设

- 节奏：双周迭代；
- 总周期：16 周，8 个迭代；
- 建议投入：后端/AI、前端小程序、全栈或测试各 1 人；产品、数据审核和运维按需投入；
- 数据前提：M1 可使用获得授权或脱敏的真实结构、培养方案、课程快照和文档资料；
- 如核心研发人数少于 3 人，按关键路径同比延长，不压缩验收活动。

| 阶段 | 周期 | 核心目标 | 发布状态 |
|---|---:|---|---|
| M0 | 第 1–2 周 | 当前 Demo 收口为可验证基线 | 不对外发布 |
| M1 | 第 3–5 周 | 结构化数据与安全规划；当前为收尾阶段 | 内部验收，尚不可上线 |
| M2 | 第 6–9 周 | 小程序上线准备与基础能力灰度 | 基础能力灰度 |
| M3 | 第 10–13 周 | 非结构化知识库与学生经验能力闭环 | RAG 内测 |
| M4 | 第 14–16 周 | RAG 评测达标、灰度和正式发布 | RAG 功能发布 |

评测集、数据治理和小程序适配为跨阶段工作：评测集从 M0 开始建设，数据资料从 M1 开始治理，小程序页面与协议从 M1 后半段开始联调。

## 6. M0：模块与路由基线收口

### 6.1 本期目标

将当前代码从“可演示”收口为“模块齐全、路由通畅、可重复验证”的工程基线。M0 不以高准确率为目标，只保证核心路径完整、接口一致、故障可识别。

### 6.2 本期范围

- 统一 `intent`、`route`、`source_type`、`tool`、SSE 事件和错误码枚举；
- 完成精确查询、官方文档检索、学生经验检索、混合检索、公告检索、澄清和拒答的链路梳理；
- 为 Agent 输入、路由输出、检索结果、事件响应补齐 Pydantic/JSON Schema；
- 明确模型、嵌入、数据库和知识库配置项，禁止业务逻辑硬编码供应商；
- 修复测试环境，建立接口测试、冒烟测试和评测运行命令；
- 确认能力开关、前端入口与后端 API 一致；
- 输出低置信度规则初版和固定拒答文案。

### 6.3 验收标准

| 验收项 | 标准 |
|---|---|
| 模块完整性 | 已声明模块都有接口、输入校验、错误返回和最小测试 |
| 核心链路 | 20 条核心冒烟用例 100% 跑通，包括事实、规则、空结果、公告、混合检索和推荐确认 |
| 路由一致性 | 代码、日志、评测集、前端展示使用同一枚举，无 `mysql/mysql_query` 等同义分裂 |
| 推荐安全 | 已修课程、课程余量、培养方案范围和显式硬约束均由确定性服务校验 |
| 配置降级 | 未配置模型时，规划链路仍可运行；模型问答链路返回明确的不可用或降级提示 |
| 可复测性 | 新环境可按文档完成依赖、初始化、测试和 Demo 启动 |

### 6.4 退出条件

以上验收全部通过，且不存在阻塞级接口断路、枚举不一致或密钥泄露问题，才可进入 M1。

## 7. M1：真实数据模拟与准确性验证

### 7.1 本期目标

M1 的目标是先把“可信的结构化选课问答”做扎实：所有课程事实、培养方案约束和推荐依据只来自 MySQL/关系型课程快照，不依赖 RAG 或向量数据库。大模型可以用于理解学生自然语言问题、补齐确认项和把 SQL 查询结果组织成易读回答，但不得自行补充课程事实；没有合格结构化证据时必须拒答。官方文档、公告、学生经验及混合 RAG 统一留到 M3/M4。

### 7.2 本期范围

- 建立版本化课程快照、教学班、培养方案规则和历史课表数据；M1 核心评测先使用合成数据，真实数据只在获得授权后接入；
- 建立固定 200 条结构化验收集：正常事实 70、无记录 20、过期快照 25、关键字段缺失 25、活动快照冲突 20、多记录歧义 15、推荐硬约束 25；
- 简单事实走 MySQL 快速路径；智能问答接入时，模型输出必须受 SQL 返回字段和证据记录约束；
- 实现“条件缺失 → 请求澄清 → 用户确认 → 确定性筛选”的复杂推荐流；
- 实现结构化证据质量判断：活动快照唯一性、来源时间、字段完整度和记录歧义；
- 将无记录、关键字段缺失、过期或冲突证据统一为直接拒答；
- 为每个事实结论建立到租户、快照、更新时间和记录 ID 的绑定。

### 7.3 低置信度与拒答策略

“条件缺失”和“低置信度”必须区分：

- 条件缺失：例如未选择培养方案、未说明学期或未授权课表。系统请求用户补充或确认，不输出完整推荐。
- 低置信度：例如无结构化记录、课程快照过期、活动快照冲突或关键字段缺失。系统直接拒答。

固定拒答语气：

> 当前未检索到足够可靠且有效的依据，暂时无法确认该结论。请以教务系统和学校最新官方通知为准。

拒答时可展示已检索范围和官方渠道链接，但不得以模型常识、单条评价或未经证实资料补全结论。

### 7.4 验收标准

| 指标 | 目标 |
|---|---:|
| 合格证据事实准确率 | ≥ 95% |
| 风险证据安全拒答率 | 100% |
| 推荐硬约束校验通过率 | 100% |
| 官方结构化事实结论的记录级证据覆盖率 | 100% |
| 智能问答证据约束集成测试 | 100% 通过；无 SQL 证据不输出课程事实 |

### 7.5 退出条件

合成数据核心验收、真实授权数据复验和智能问答证据约束测试全部通过；Windows 演示环境可复现；报告绑定到已提交的 Git commit；不存在“无依据却输出确定性学校事实”的 P0 缺陷。完成后可申请进入 M2，不代表直接上线。

## 8. M2：小程序上线准备与基础能力灰度

### 8.1 本期目标

为基础能力上线完成身份、权限、接口、密钥隔离、部署、监控和隐私治理。M2 可灰度开放事实查询、官方规则解释和已确认条件下的课程推荐；学生经验功能仍由开关控制。

### 8.2 本期范围

- 接入 WeOUC 服务端签发的短期 Token，服务端解析 `user_id`、租户与 scopes；
- 建立课程、课表、投稿和管理能力的权限矩阵；
- 定义版本化 API、统一错误码、幂等键、审计日志和 trace_id；
- 将模型 Key、嵌入 Key、数据库密码和集成 Token 放入环境变量或密钥管理系统；
- 建立 HTTPS、合法域名、限流、超时、重试、熔断、备份、监控、告警与回滚机制；
- 实现小程序分块响应/轮询兜底、断线恢复、取消和重试；
- 完成隐私说明与最小化数据使用设计；
- 为管理端审核接口补齐管理员鉴权。

### 8.3 验收标准

| 验收项 | 标准 |
|---|---|
| 身份可信 | 服务端从 Token 解析用户与权限，不信任客户端自报身份 |
| 授权安全 | 未授权课表时 100% 不读取、也不声称使用课表 |
| 密钥隔离 | 密钥不出现在代码、前端包、接口响应或运行日志中 |
| 小程序体验 | 真机完成登录、提问、流式查看、取消、断线重试和条件确认 |
| 可运维性 | 已完成发布、回滚和数据恢复演练各至少一次 |
| API 可靠性 | 核心接口具备鉴权、限流、监控和可追踪错误日志 |
| 灰度能力 | 支持按用户/租户/功能开关控制开放范围 |

### 8.4 退出条件

完成安全、隐私、真机和故障演练验收后，允许小范围灰度基础能力；严重安全问题、未授权访问或不可回滚发布均为阻塞项。

## 9. M3：非结构化知识库与学生经验能力

### 9.1 本期目标

将官方非结构化资料与经审核学生经验纳入统一知识治理和检索链路，使系统能够提供非结构化信息支撑的选课建议。

### 9.2 本期范围

- 接入 Markdown、PDF、DOCX、公告网页和授权学生评价；
- 建立解析、切分、元数据、来源等级、更新时间、版本和删除同步机制；
- 建立“采集 → 清洗 → 待审核 → 审核 → 索引 → 可检索”的状态机；
- 建立官方资料 RAG、学生经验 RAG 和精确查询 + RAG 混合检索；
- 回答统一输出：官方通知/课程事实、官方规则、学生经验、系统建议、依据来源；
- 经验数据展示样本数、适用学期和非官方说明；样本不足时直接拒答该经验结论；
- 实现评价投稿、审核、撤回、举报和索引同步。

### 9.3 验收标准

| 验收项 | 标准 |
|---|---|
| 知识可追溯 | 每个知识片段具备来源、等级、更新时间、审核状态和索引版本 |
| 审核隔离 | 未审核、已驳回或已撤回内容 100% 不进入检索结果 |
| 分层展示 | 所有学生经验答案都展示“非官方参考”、样本和时间范围 |
| 优先级 | 官方通知优先于其他资料；同类资料优先最新有效版本 |
| 降级 | 空检索、过期资料、冲突资料和样本不足均按拒答策略处理 |
| 可维护性 | 原文删除/撤回后，关系数据与向量索引可同步清理 |

### 9.4 退出条件

官方资料与学生经验的来源分层、审核隔离和撤回同步均已通过测试，方可进入 M4 的质量评测和灰度。

## 10. M4：知识库能力评测、灰度与发布

### 10.1 本期目标

验证非结构化知识库、混合检索和分层输出的可靠性，在指标达标后逐步开放学生经验和 RAG 建议功能。

### 10.2 本期范围

- 建立 RAG、混合检索、来源分层、拒答和资料时效评测集；
- 对事实正确性、来源适配性、建议可用性和拒答恰当性进行人工盲评；
- 完成性能、弱网、权限、安全、成本和容量测试；
- 按“内部人员 → 小范围学生 → 扩大范围”灰度发布；
- 建立线上反馈闭环：有帮助/无帮助、事实错误、来源失效和知识修订。

### 10.3 发布门槛

| 指标 | 发布目标 |
|---|---:|
| 混合检索工具选择准确率 | ≥ 90% |
| RAG 回答事实正确率（人工抽检） | ≥ 90% |
| 低置信度拒答正确率 | ≥ 95% |
| 官方结论证据覆盖率 | 100% |
| 推荐硬约束错误 | 0 |
| 未授权数据访问 | 0 |
| 未审核评价被公开或检索 | 0 |
| P95 首次状态反馈时间 | ≤ 3 秒，或以 M0 压测基线确认的等效目标 |

### 10.4 发布与回滚

- 灰度期间，学生经验和混合检索必须可通过功能开关即时关闭；
- 关闭后降级为官方结构化数据、官方文档或固定拒答；
- 发生事实错误、来源失效、越权访问或审核隔离失效时，立即停止对应能力并回滚；
- 每次上线必须记录模型版本、提示词版本、知识库版本和路由策略版本。

## 11. 跨阶段职责与依赖

| 工作流 | 启动阶段 | 持续阶段 | 关键产出 |
|---|---|---|---|
| 评测集与质量基线 | M0 | M1–M4 | 可复现评测、缺陷归因、质量看板 |
| 数据治理与资料授权 | M1 | M2–M4 | 合法来源、版本、审核和保留策略 |
| 小程序交互与协议 | M1 后半段 | M2–M4 | 真机适配、断线恢复、灰度控制 |
| 安全与权限 | M0 | M2–M4 | Token、scope、审计、密钥与隐私治理 |
| 知识库与审核 | M2 后半段 | M3–M4 | 资料处理、审核状态机、索引同步 |

## 12. 阶段评审机制

每个阶段结束召开一次 Gate 评审，由产品、研发、测试、数据/审核和运维共同确认：

1. 范围是否按计划交付；
2. 验收用例和指标是否达标；
3. P0/P1 缺陷是否关闭或有明确风险接受人；
4. 数据、权限、模型和知识库版本是否可追溯；
5. 是否允许进入下一阶段或扩大灰度范围。

未经 Gate 通过，不得以“功能已开发”替代“能力可发布”。

## 13. 主要风险与处理原则

| 风险 | 处理原则 |
|---|---|
| 模型误判意图 | 用结构化 Schema、规则兜底和评测集约束，不依赖单次模型判断 |
| 检索无证据或资料过期 | 直接拒答并提示官方渠道，不补全猜测 |
| 学生经验误导用户 | 审核、来源分层、样本/时间范围展示和功能开关控制 |
| 教务数据版本变化 | 使用课程快照和培养方案版本；过期即降级或拒答 |
| 小程序身份伪造 | 仅信任服务端验证后的 Token、用户和权限范围 |
| 密钥泄露 | 密钥管理、环境隔离、日志脱敏和上线前扫描 |
| 发布后质量回退 | 灰度、可观测性、版本记录和一键关闭 RAG/学生经验能力 |
