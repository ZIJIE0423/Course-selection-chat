# WeOUC 选课规划第一阶段标准接口契约

## 1. 模块能力

H5 启动后先调用：

```http
GET /api/v1/capabilities
```

第一阶段默认开启 `course_planning` 和 `academic_history_import`，关闭课程反馈、学生评价 RAG、课表冲突。

## 2. 小程序/WeOUC 服务端数据导入

服务端请求使用：

```http
X-Integration-Token: <INTEGRATION_TOKEN>
```

### 导入培养方案版本

```http
POST /api/v1/integrations/programme-versions
Content-Type: application/json
```

培养方案规则支持：

- `required`: 必修课程；
- `elective_pool`: 允许选择的课程或课程类别；
- `credit_requirement`: 某课程类别的学分要求（第一阶段保留数据，详细毕业进度计算后续扩展）。

### 导入当前学期课程快照

```http
POST /api/v1/integrations/course-offering-snapshots
Content-Type: application/json
```

同一 `tenant_id + snapshot_id` 的相同内容重复提交会返回幂等成功；内容不同返回 `409`，调用方必须生成新快照 ID。新快照导入后，同租户、同学期的旧活动快照自动标记为 `superseded`。

## 3. 历史课表

### 上传文件

```http
POST /api/v1/academic-history/imports
Content-Type: multipart/form-data
X-Tenant-Id: weouc
X-User-Id: <anonymous_business_user_id>
```

支持 CSV、TSV、XLSX、JSON、TXT、Markdown、文本型 PDF。图片需要配置 `HISTORY_OCR_ENDPOINT`；未配置时返回 `422`，H5 应切换到手工补录。

所有导入课程初始状态为：

```json
{
  "completion_status": "assumed_passed",
  "status_source": "historical_timetable_default",
  "confirmed_by_user": false
}
```

### 确认解析结果

```http
POST /api/v1/academic-history/imports/{import_id}/confirm
```

用户可在确认请求中修正课程名称、课程号、学期、学分或状态。确认前的记录不得用于生成确定性毕业结论。

### 查询已保存记录

```http
GET /api/v1/academic-history/records
X-Tenant-Id: weouc
X-User-Id: <anonymous_business_user_id>
```

## 4. 学生档案与培养方案确认

```http
GET /api/v1/student/programmes?tenant_id=weouc&grade=2024级&major=计算机科学与技术
GET /api/v1/student/profile?tenant_id=weouc&user_id=<user_id>
PUT /api/v1/student/profile
```

`PUT` 请求必须提交 `tenant_id`、`user_id`、年级 `grade`、学院 `department`、专业 `major` 和 `programme_version_id`。浏览器不再把这些业务资料存入 `localStorage`；正式接入时 `tenant_id/user_id` 应由公众号登录 Ticket 在服务端解析。

## 5. 选课规划

### 获取当前规划上下文

```http
GET /api/v1/planning/context?tenant_id=weouc&user_id=<user_id>
```

H5 在创建会话前调用该接口，获取当前活动课程快照和学生适用的培养方案版本。前端不得写死数据库内的 `snapshot_id` 或 `programme_version_id`。若学生档案尚未完成后端绑定，接口会临时返回租户下最新活动培养方案，并将 `confirmed` 标记为 `false`；H5 必须提示学生正式使用前完成确认。

### 创建规划会话

```http
POST /api/v1/planning/sessions
Content-Type: application/json
```

请求包含 `tenant_id`、`user_id`、数据库内 `snapshot_id`、`programme_version_id` 和自然语言 `query`。

系统返回：

- 结构化约束；
- 结构化偏好；
- 当前阶段不支持的偏好；
- `awaiting_confirmation`、`awaiting_history_confirmation` 或其他状态。

### 确认需求并执行筛选

```http
POST /api/v1/planning/sessions/{session_id}/requirements/confirm
```

确定性筛选顺序：

1. 活动课程快照；
2. 培养方案选修池；
3. 排除 `assumed_passed/passed` 历史课程；
4. 排除无余量课程；
5. 应用校区、星期、课程类别、教师和学分等硬约束；
6. 应用支持的软偏好并排序；
7. 返回来源、快照版本和风险提示。

### 确认对话中的历史课程纠正

当自然语言包含“没过、挂科、退课、重修”等表达时，创建规划会话只返回候选记录，不直接修改。用户确认后调用：

```http
POST /api/v1/planning/history-corrections/confirm
```

## 6. 课程详情

推荐结果包含活动课程 `offering_id`，H5 用它读取同一快照中的真实详情：

```http
GET /api/v1/catalog/offerings/{offering_id}?tenant_id=weouc&user_id=<user_id>
```

响应包含课程安排、教师、校区、容量、余量、快照依据、培养方案关系和历史修读状态。调用方不能再根据课程号读取前端静态 `data.js`。

## 7. 生产接入前的安全要求

- 必须配置 `INTEGRATION_TOKEN`；
- `X-User-Id` 当前是内部业务标识接口，正式 H5 接入时应由一次性登录 Ticket 换取，不能信任浏览器直接声明的身份；
- 上传文件需要在网关或文件服务增加病毒扫描；
- OCR 服务必须明确数据处理范围和保留策略；
- 课程快照、培养方案和历史修读数据均需按 `tenant_id` 隔离。
